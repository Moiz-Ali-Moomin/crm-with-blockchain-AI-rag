Prisma Migration Guide

Everything you need to know about creating, applying, and managing database migrations for the CRM SaaS platform — from local development to production deployment.

📁 crm-backend/prisma/migrations/ 🔧 Prisma ORM v5 🐘 PostgreSQL 16 + pgvector 🕐 Last updated: 2026-04-07
📋

Overview

A migration is a versioned SQL script that describes a change to the database schema — adding a table, a column, an index, or an enum value. Migrations are stored in git alongside the code so that every environment (local, staging, production) can reach exactly the same DB state by running the same sequence of scripts.

In this project, Prisma Migrate handles migration generation automatically. You change schema.prisma, run one command, and Prisma generates the SQL diff and records it as an immutable file. You never write raw SQL by hand for schema changes (except for pgvector-specific DDL that Prisma cannot express natively).

Why not just run ALTER TABLE directly on the server?

Concern ❌ Direct SQL on server ✅ Prisma Migrations
Reproducibility Done once, forgotten Committed to git, runs everywhere
Tracking No record of what changed or when _prisma_migrations table tracks every run
Team sync Other devs' DBs drift out of sync prisma migrate dev syncs everyone
CI/CD Manual step on every deploy prisma migrate deploy runs automatically
Rollback Guesswork Planned down-migration scripts

How It Works

Prisma compares your current schema.prisma against the last migration file to compute the SQL diff. It then writes a new .sql file and applies it.

The _prisma_migrations tracking table

PostgreSQL

Every time a migration runs, Prisma inserts a row into _prisma_migrations. This is how prisma migrate deploy knows which scripts have already been applied and which are pending.

SQL — inspect applied migrationspsql
SELECT migration_name, finished_at, applied_steps_count
FROM _prisma_migrations
ORDER BY started_at DESC;

If a migration file is deleted from disk but still in the table, migrate deploy will error. Never delete migration files from the repo.

Migration file naming convention

Files are stored at prisma/migrations/{timestamp}_{name}/migration.sql

Example directory structurebash
prisma/migrations/
├── 20260404101255_init/
│   └── migration.sql          # ← all 22 tables created
├── 20260405000001_add_ai_embeddings_and_blockchain/
│   └── migration.sql          # ← ai_embeddings + blockchain_records
├── 20260406000001_add_paypal_subscription_id/
│   └── migration.sql          # ← paypalSubscriptionId column
└── migration_lock.toml        # ← DO NOT edit this file

The timestamp prefix guarantees migrations run in the correct order. The name portion is what you provide with --name.

⌨️

Commands

migrate dev dev only

Creates a new migration from schema diff and applies it locally. Also regenerates the Prisma client.

npx prisma migrate dev
  --name add_sla_deadline
migrate deploy prod safe

Applies all pending migrations in order. Never generates new SQL. Safe to run on production.

npx prisma migrate deploy
migrate reset ⚠ wipes DB

Drops the entire database, re-applies all migrations from scratch. Dev only. Never run on prod.

npx prisma migrate reset
generate

Regenerates the Prisma client from schema without touching the DB. Run after pulling schema changes.

npx prisma generate
migrate status

Shows which migrations are applied vs pending. Useful for debugging environment drift.

npx prisma migrate status
💡
All commands run from crm-backend/

Always cd crm-backend first. The DATABASE_URL env var must be set (check .env).

Dev → Production Workflow

1

Edit schema.prisma locally

Add a model, a field, an index, or an enum value. Don't touch migration files directly.

2

Run prisma migrate dev --name <description>

Prisma diffs the schema, generates SQL, applies it to your local DB, and regenerates the client. A new folder appears in prisma/migrations/.

crm-backend/bash
npx prisma migrate dev --name add_contact_linkedin_score
3

Commit both files together

Always commit schema.prisma and the generated migration.sql in the same commit. Never commit one without the other.

gitbash
git add prisma/schema.prisma prisma/migrations/
git commit -m "feat: add linkedin_score to contacts"
4

Push to main — CI/CD takes over

The GitHub Actions CD pipeline SSHes into the Hetzner server, runs git pull, then automatically executes:

.github/workflows/cd.ymlbash
npx prisma generate
npx prisma migrate deploy   # ← applies your new migration
npm run build
node dist/src/main.js &
5

Verify on server

SSH in and confirm the migration was applied:

serverbash
cd /opt/nexus-crm/crm-backend
npx prisma migrate status

Creating a New Migration — Step by Step

Worked example: adding a slaDeadline field to the Ticket model.

Step 1 — Edit schema.prisma

Add the new field to the relevant model. Use ? (optional) for new columns on existing tables so existing rows aren't forced to supply a value.

crm-backend/prisma/schema.prismaprisma
model Ticket {
  id          String         @id @default(uuid())
  // ... existing fields ...
  resolvedAt  DateTime?
  closedAt    DateTime?
  slaDeadline DateTime?   // ← new field
  tags        String[]       @default([])
}

Step 2 — Generate + apply the migration

crm-backend/bash
npx prisma migrate dev --name add_ticket_sla_deadline

Prisma generates this file automatically at prisma/migrations/20260407xxxxxx_add_ticket_sla_deadline/migration.sql:

generated migration.sqlsql
-- AlterTable
ALTER TABLE "tickets"
  ADD COLUMN "slaDeadline" TIMESTAMP(3);

Step 3 — Add a pgvector raw column (special case)

Prisma cannot express the vector type natively. For vector columns, write the migration SQL manually and mark the field with a comment in the schema:

manual migration.sqlsql
-- pgvector column — not expressible in Prisma DSL
ALTER TABLE "ai_embeddings"
  ADD COLUMN "embedding" vector(1536);

CREATE INDEX IF NOT EXISTS "ai_embeddings_embedding_idx"
  ON "ai_embeddings"
  USING ivfflat ("embedding" vector_cosine_ops)
  WITH (lists = 100);

Step 4 — Commit both files

gitbash
git add prisma/schema.prisma
git add prisma/migrations/
git commit -m "feat: add slaDeadline to tickets"
git push origin main   # triggers CD pipeline

Gotchas & Rules

🚫
Never delete or edit a migration file that has been applied to production

Prisma stores a checksum of each file. If the file changes, migrate deploy will refuse to run and the deploy will fail.

⚠️
New columns on existing tables must be nullable or have a default

If you add NOT NULL without a default, PostgreSQL will error because existing rows have no value. Always use ? in Prisma (which maps to nullable SQL) for additive changes.

⚠️
IVFFlat index cannot use CONCURRENTLY inside Prisma transactions

Prisma wraps migrations in a transaction. CREATE INDEX CONCURRENTLY is not allowed inside a transaction — it will error. Use plain CREATE INDEX IF NOT EXISTS instead (as done in 20260405000001).

⚠️
Prisma keeps camelCase column names in PostgreSQL

Columns are named "wonAt", "tenantId", "ownerId" — not snake_case. Raw $queryRaw statements must quote them exactly. Using won_at or tenant_id will cause a 500 error.

💡
Use pgvector/pgvector:pg16 Docker image — not postgres:16-alpine

The standard alpine image does not include the pgvector extension. The CREATE EXTENSION vector statement in the AI migration will fail unless you use the pgvector image.

💡
Always run prisma generate after pulling schema changes

The Prisma client is generated code — it is not committed to git. After git pull, run npx prisma generate before starting the backend or it will use a stale client.

↩️

Rollback Strategy

Prisma Migrate does not have built-in rollback (unlike some other ORMs). The recommended strategy is to write a forward-fix migration that undoes the change.

1

Revert schema.prisma to the previous state

Remove the field, table, or enum value you want to undo.

2

Run prisma migrate dev --name revert_<description>

Prisma generates the inverse SQL (e.g. DROP COLUMN) as a new migration file.

3

Commit and push

The CD pipeline applies the revert migration on the server automatically.

🚨
Dropping a column is destructive and permanent

Always check if the column contains production data before running a drop migration. Consider using a two-phase approach: first deploy code that stops using the column, then drop it in a separate migration after confirming no issues.

CI/CD Integration

Migrations run automatically on every push to main via the GitHub Actions CD pipeline at .github/workflows/cd.yml.

What the deploy script does

.github/workflows/cd.yml
Deploy sequence on Hetzner (37.27.41.226)bash
cd /opt/nexus-crm

# 1. Pull latest code (includes new migration files)
git pull origin main

# 2. Install deps
npm ci --prefer-offline

# 3. Regenerate Prisma client from updated schema
npx prisma generate

# 4. Apply any pending migrations — idempotent, safe to re-run
npx prisma migrate deploy

# 5. Build and restart
npm run build
kill $(fuser 3001/tcp 2>/dev/null) || true
nohup node dist/src/main.js > /tmp/crm-api.log 2>&1 &

✅
prisma migrate deploy is idempotent

It only runs migrations that haven't been applied yet. Re-running it on a fully up-to-date DB does nothing. It is safe to run on every deploy even if there are no new migrations.
